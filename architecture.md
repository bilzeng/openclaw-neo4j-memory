# OpenClaw 知识图谱记忆插件 (Neo4j-Memory V4) 架构解析

这是针对 `neo4j-memory` 插件最新 V4 RAG 架构（融合了大语言模型事实抽取与跨智能体共享技术）的深度解析文档。本文档将通俗易懂地说明整个插件的工作原理、每个文件的具体职责，以及系统是如何实现全自动“听取对话 -> 拆解知识 -> 全局共享”的。

---

## 💡 一、核心原理解析：读写分离的“双钩子”架构

这个插件的魔法在于，它极其巧妙地在 OpenClaw 的通信主干道上挂载了**一前一后两个钩子 (Hooks)**，形成了一个“对话前看记忆，对话后记笔记”的闭环。

### 1. 读记忆：对话触发前（RAG 检索注入）
当你在飞书或者终端发出一句话（例如：“我爱吃什么？”）的瞬间，这句话并没有立刻发给大模型。
它首先被插件的第一个拦截器接管。系统会飞速提取你这句话里的“关键词”，去 Neo4j 图谱数据库里翻找相关的“事实记录”。如果找到了（如 `[Boss] -[爱吃]-> [苹果]`），插件会**神不知鬼不觉**地把这段话强行塞进你原来的提示词（Prompt）里，变成类似：
> “【图谱记忆：Boss爱吃苹果】。真实提问：我爱吃什么？”

有了这层记忆外衣，无论哪个智能体接到了这句话，都能立刻装作“过目不忘”的样子流利对答。

### 2. 写记忆：对话结束后（SPO 三元组拆解）
当 AI 顺利回答完你的问题之后，第二个钩子会被触发。插件会在后台派出一个专门的“知识提炼大模型（Qwen）”。
这个独立大模型会通读你们俩刚才的聊天记录，像人类做思维导图一样，把冗长的对话**原子化、碎片化拆解**。它会严格提炼出 `主语 (Subject)` - `关系动词 (Predicate)` -> `客体 (Object)`，即 SPO 三元组。随后，这些小巧精准的节点会自动写入 Neo4j 数据库。

## 📁 二、核心文件与作用清单

整个系统是由轻量级的 `Node.js` 路由网关，和沉重的 `Python` AI 处理脚本配合完成的。

### 1. `index.js` (插件总闸门)
该文件是整个插件系统的**大管家与挂载点**。
- **`api.intercept('message:before')`**：真正的**提问前拦截器**。它截获用户的原生问题 `context.content`，然后去调用外部 Python 脚本。如果 Python 找到了相关图谱记忆，它会覆写 `context.content`，完成 RAG 记忆注入。（*注：必须使用 intercept 而不是 on 才能实际改变发送给大模型的文字流*）。
- **`api.on('agent_end')`**：真正的**事后记录器**。当对话完美落幕后，它提取出用户说了什么、AI回答了什么，然后把这些文本通过异步 `spawn` 给后台 Python 子进程处理，毫不影响主系统的流畅性。

### 2. `python/retrieve.py` (知识猎犬)
它是负责“查日记”的侦探程序。
- **关键词抽取**：收到一句话时，它会首选调用阿里云 DashScope 大模型提取中心词。
- **极限降级算法**：如果你在弱网环境，API 超时了，它会利用本地“过滤无用词汇（如：的、是、怎么、我、你）”的算法，把“我爱吃什么”降级切分为“爱吃”。
- **双向高容错 Cypher**：它在查询 Neo4j 数据库时，使用了增强的 `CONTAINS` 匹配算法。它允许：**查询词包含在图谱关系中（比如搜“苹果”，匹配出“大苹果”），也允许图谱关系包含在查询词中（比如搜“我爱吃”，精确命中图谱中的“爱吃”动作）**。保证了极高的检索召回率。

### 3. `python/neo4j_graph_memory.py` (知识锻造炉)
插件里最硬核的文件，所有大篇幅废话到此全部变成“图谱神经元”。
- **`_extract_triplets_with_llm`**：图谱最重要的方法。它通过极严苛的 Prompt 设计，要求 LLM 把句子拆散成“字数小于15个字”的迷你三元组（例如：把长句直接拆成：`[Boss] -[提问]-> [模块设计]`, `[小程序] -[包含]-> [课文点读]`）。它甚至能根据上下文，推测被人类否决了的错误技术方案，并入库做防坑标记！
- **`insert_after_chat`**：执行原子级入库。它动态构建 Cypher 语句，强行把大模型解析出来的不确定动词（比如“擅长”、“讨厌”）转化为图谱中的实时边（Relationship）。同时还会打上时间戳并跟原始聊天记录做溯源绑定。

### 4. `python/hook.py` (通信桥梁)
因为 `Node.js` 不方便直接执行极度复杂的机器学习逻辑和 Python 原生库，这里提供了一个过渡脚本。它主要作用就是接收 `index.js` 传来的命令行参数，并将其转化为对 `neo4j_graph_memory.py` 中 `insert_after_chat` 函数的标准调用。

## 🚀 三、为什么现在如此强大？

在近期的 V4 更新中，系统打通了最后的任督二脉：
1. **全局共享机制**：以前你的记录可能只存在主 Agent 脑子里。现在，`message:before` 拦截器设置在整个 OpenClaw 流量池的最上游。任何 Agent 被提问，不管它是后端架构师还是产品经理，都会被迫先经过图谱检索，因此全体 Agent 会自动共享同一套“常识库”。
2. **容错与鲁棒性**：不管是提取事实因为大模型超时失败，还是由于查询词没有完全匹配，系统都有多级保护垫。它最终依然能通过精确的局部切词和截断，像人类直觉一样把“你想起了什么”的拼图完整凑出。
